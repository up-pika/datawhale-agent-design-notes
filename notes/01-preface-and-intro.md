## 1. 题记：一位「模式思维者」的告白

讲义强调：比起孤立的事实，更应关注**知识如何被组织成可理解的整体**——答案背后的**结构性前提**，往往比结论本身更值得记住。

围绕「模式」长期训练，会沉淀两类能力：

| 能力 | 含义 |
|------|------|
| **模式发现** | 从复杂、看似杂乱的现象中，抽出反复出现的共同结构 |
| **模式生成** | 在深入理解既有模式的基础上，自主搭建新的框架与表达语言 |

一句话：**不只为「知道什么」，更为「看清结构为何成立」。**

## 2. 序言：为什么需要一场「重塑」

### 2.1 核心判断

- **黄佳**：智能体（Agent）不只是又一个新模型，而是**一种新的物种**——它让软件第一次真正拥有 **「意图」** 与 **「时间」** 这两个维度。
- **重塑的目标**：把系统从「被动执行的代码」，变成能 **感知、反思、协作** 的 **主动行动者**。

### 2.2 传统软件工程与「不确定性」

结构化编程、面向对象、微服务等，在**目标相对稳定、环境相对可控**时很有效；当前复杂度很大程度上来自 **不确定性**。

**Agent 时代的四个特征**（讲义归纳）：

1. **开放的任务目标**：需求在对话与协作中持续生成，而非一次性冻结。
2. **动态环境**：外部信息持续流入，世界状态不断变化。
3. **概率性决策**：路径依赖权衡与反思，难以用固定分支穷举。
4. **世界影响回路**：输出会作用于真实世界，世界再反过来塑造下一轮输入。

**结论**：经典的 **输入 → 处理 → 输出** 范式不够用了；这一时代需要类似经典 GoF 那样——**可沟通、可继承、可演化** 的 **Agent 设计模式** 语言。

## 3. 设计模式在 AI 时代为何重新变得重要

### 3.1 价值迁移

- 从「**模板化代码**」→「**可复用的思维路径**」。
- 从「**代码复用**」→「**决策复用**」：传统模式复用类与关系；Agent 模式复用 **感知方式、推理逻辑、行动策略**，在不确定环境中争取 **可预期的行为**。
- 从「**静态结构**」→「**动态闭环**」。讲义中的闭环表述包括但不限于：
  - **感知 → 记忆 → 推理 → 决策**；
  - **感知 → 行动 → 反馈 → 反思 → 再行动**；
  - **策略 → 行动 → 反馈 → 反思 → 再行动**（并强调需考虑触发条件、阈值与 **护栏**）。

### 3.2 四个「转向」与设计模式作为基础设施

1. **从局部实现到跨边界协作**：与工具、服务、其他 Agent 协同；模式固化 **意图表达、协议交互、记忆共享**，把边界变成可组合模块。
2. **从代码可读到行为可解释**：不仅「能跑」，还要能说明白；用 **意图—结构—结果** 链条支撑可追溯性。
3. **从工程实践到组织对齐**：模式成为团队讨论「AI 该如何行动」的 **共同语言**，便于协作与治理。
4. **模式即基础设施**：在决策、记忆、反思、协作等层面减少重复造轮子。

## 4. 从「对象」到「智能体」：系统边界的变化

| 维度 | 对象（OOP 视角） | 智能体（Agent 视角） |
|------|------------------|----------------------|
| **目标** | 有状态与方法，但 **无内在目标**；被动等待调用 | **带着目标进入世界**；主动感知、规划、行动并调整策略 |
| **生命周期** | 被动调用 | **持续运行**（或长周期、事件驱动下的主动运转） |
| **接口** | 方法签名 | **意图协议**（要达成什么，而不仅是调哪个函数） |
| **状态** | 多强调进程内局部状态 | 状态 **嵌入世界**：环境、其他主体、**时间** 都是状态的一部分 |

### 4.1 用「主动视角」重构图谱系结构

讲义建议把常见实现概念映射为 Agent 架构语言：

| 传统实现视角 | Agent / 认知工程视角 |
|--------------|----------------------|
| 输入 | **感知** |
| 中间状态 | **记忆与上下文** |
| 算法 / 调用 | **推理与规划** |
| 函数执行 | **行动与协作** |
| 日志 | **反思与自我改进** |

### 4.2 小结句（讲义精神）

- **面向对象仍然有效**，但在跨出纯代码、进入真实世界时，更像一种 **「子语言」**。
- **Agent 设计模式** 被定位为这一时代的 **「母语」**：当系统跨越代码边界、步入真实世界，**被动的对象要让位于主动的行动者**。

## 6. 五条核心设计原则（实践抓手）

1. **目标优先**：从「要达成什么」出发，而不是从「手头有哪些工具可调用」出发。
2. **上下文为王**：提示词可见，但真正行为往往由 **上下文工程** 与 **记忆管理** 决定。
3. **显式反馈**：把反馈、反思、改进做成 **内置能力**，而不是事后打补丁。
4. **渐进自治**：在 **安全护栏** 内逐步提高自主性，同时保留人的 **介入与解释权**。
5. **对齐与护栏**：伦理、安全、合规应 **写进结构与协议**，而非仅靠运气或事后补救。

**目标读者**（讲义自述）：

- **工程师 / 架构师**：把复杂智能系统看成可复用的「组件与模式」组合。
- **产品 / 运营**：掌握与工程对话的「共同语言」。
- **研究者 / 教育者**：观察理论如何落地为可教、可复用的设计模式。

**技术叙事的时间尺度**（讲义提法）：

- 过去约百年：**让机器会做事**
- 过去约十年：**让机器会说话**
- 未来约十年：**让机器会处世**（与社会、制度化协作共处）

设计模式被定位为描述 **系统意图、接口与边界** 的 **新语言**，推动 Agent 成为可与人类长期协作的基础设施。

## 8. 自测与复习（可把答案写在本节下方）

- 用你自己的话解释：**「决策复用」与「代码复用」差在哪里？**
- 画出或写出一种你理解的 **闭环**：从感知到协作，哪一环最容易在工程上被省略？为什么？
- 五条原则里，你认为当前团队最薄弱的是哪一条？对应一个可落地的改进小动作。

### 自测作答

**1. 决策复用 vs 代码复用**

代码复用复用的是已经写好的「结构」——类、函数、模块，在输入输出相对稳定时反复调用即可。决策复用复用的是在不确定环境里「怎么想、怎么做」的那条路：怎么感知、怎么推理、选什么策略、何时停手与反思。前者省的是重复实现，后者争取的是在开放目标与动态世界里行为仍然可预期、可解释、可治理。

**2. 闭环与最容易被省略的一环**

我用的闭环：**感知 → 记忆 → 推理 → 行动 →（外部）反馈 → 反思 → 再行动**；若强调协作，可在行动前后接上 **与其他 Agent / 人的意图对齐与交接**。

最容易先被省略的是 **反思与显式反馈**：工程上往往「能答出来」就交付，日志有，但没有把失败样例、用户纠正、工具报错 **结构化地** 接回记忆与策略；触发条件、阈值、护栏也常来不及做。其次是 **协作协议**（意图、边界、责任），单体 demo 时最容易延后。

**3. 最薄弱的原则与一个小动作**

当前我们更薄弱的是 **显式反馈**：问题能修掉，但「错因—改法」若只留在 PR 描述里，很难沉淀成可复用的模式与评测。

**小动作（与日常修 bug 对齐）**：每次定位并修复一个问题时，顺手记一条 **bug 文件**（或等价工单 / 笔记），与代码变更绑定。建议在文件里固定几项，方便日后归类和做回归样例：

- **现象**（可附用户原话或对话片段）
- **失败类型**（幻觉 / 工具误用 / 上下文丢失 / 合规与护栏等）
- **根因与改法**（改提示、加校验、补记忆、收紧工具权限等）
- **是否纳入评测或回归**（是 / 否；若否，简短说明原因）

若团队人多或需要和产品对齐「什么叫错」，可在此基础上 **按月** 从 bug 文件里抽几条过会；个人或小团队时，只把 bug 文件写规范，通常就够支撑「显式反馈」这一条原则。
